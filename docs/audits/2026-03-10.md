# Ruriko Vision Audit — 2026-03-10

> Consolidated from two independent gap analyses (one by ChatGPT, one by Claude)
> run against the full codebase, docs, TODO.md, CHANGELOG.md, and all templates.

---

## Executive Summary

Ruriko is **~80% toward the stated vision**. The core architecture — deterministic control plane, immutable agent runtime, policy engine, secret isolation, approval gates, auditable inter-agent communication — is complete and well-tested. The remaining 20% is **UX abstraction**: making the secure infrastructure feel effortless through natural conversation (conversational Gosuto authoring, conversational plan review and modification, and workflow expressiveness).

Both analyses converge on the same priority order:
1. Finish R6 (workflow engine hardening + live multi-cycle verification)
2. Build R17 (template variables + NLP extraction + **conversational plan review and modification**)
3. Close R7 (observability, circuit breakers, rate limiting)
4. Implement R18 (Gitai-side conversation memory + prompt injection defense)

## Vision-to-Reality Mapping

### 1. Self-hosted Matrix ✅ Complete (10/10)

Tuwunel (Conduit fork) is bundled, federation OFF, registration OFF. `docker compose up -d` gives a private homeserver. Ruriko and agents connect via mautrix-go. Admin-room model keeps control separate from personal chat. Multiple client recommendations documented (Element, Cinny, gomuks).

### 2. NLP Translation Layer with Distrust ✅ Complete (10/10)

The NLP pipeline (`internal/ruriko/nlp/`) translates natural language to structured intents with a three-tier confidence gate (≥0.8 proceed with mutation confirmation, ≥0.5 confirm interpretation, <0.5 refuse). Cron expressions validated after LLM output. Agent IDs sanitised. **Topology mutations blocked entirely from NLP path** — only explicit commands can change the agent mesh. Failed commands trigger re-query with error context (max 2 retries). The NLP layer proposes; the deterministic router decides.

### 3. Deterministic Security Policy ✅ Complete (10/10)

Policy engine (`internal/gitai/policy/engine.go`): first-match-wins capability rules, glob matching on MCP server and tool name, **default deny**. Three outcomes: Allow, RequireApproval, Deny. Every tool call — LLM-triggered or workflow-triggered — passes through the same policy check. Trust boundary checks on room and sender before any processing. The 12 invariants in `docs/invariants.md` are enforced in code, not aspirational.

### 4. Ruriko as Deterministic Control Plane ✅ Complete (10/10)

Command router (`internal/ruriko/commands/router.go`) is a deterministic parser — no LLM decides lifecycle, secrets, or policy operations. Six categories of operations require explicit human approval: agent deletion/disabling, secret deletion/rotation, Gosuto set/rollback.

### 5. Separation of Concerns ✅ Complete (9/10)

Three canonical agents as Gosuto templates (Saito: cron coordinator, Kairo: finance analysis, Kumo: news/search) plus three generic templates (cron-agent, browser-agent, email-agent). Each agent runs in its own container with its own Matrix account, policy, and secrets. Peer-to-peer communication over Matrix; Ruriko provisions but does not relay.

### 6. Immutable Runtime + Mutable Program ✅ Complete (10/10)

**Gitai**: single Go binary, cannot modify itself, config, or privileges. Behavior driven entirely by Gosuto YAML received via ACP. **Gosuto**: versioned, SHA256-hashed, diffable, auditable; stored in SQLite with version numbers; rollback supported; schema-validated before application. Three-layer model (Policy > Instructions > Persona) is enforced.

### 7. Embedded Tools + Event Triggers ✅ Complete (9/10)

**Built-in tools:** `matrix.send_message`, `schedule.upsert`/`disable`/`list`. **MCP:** declared in Gosuto, supervised by Gitai; Gitai Docker image includes Node.js/npm for npx-based MCP servers. **Gateways:** cron (DB-backed, 5-field + `@every`), webhook (HMAC-SHA256 authenticated, payload summarisation), external binaries (e.g., `ruriko-gw-imap`, supervised same as MCP).

### 8. Conversational Workflow Creation ⚠️ Partial (5/10)

**Works:** "Set up Saito and Kumo" → multi-step plan decomposition, step-by-step approval, canonical role knowledge from template metadata, cron extraction from NL, ambiguity triggers clarification.

**Doesn't work yet:**
- Can't describe an arbitrary new workflow and have Ruriko generate a Gosuto config for it.
- Template variable customisation at provision time not complete (R17).
- Can't conversationally update an existing agent's workflow steps.

**Root cause:** All primitives exist (versioning, ACP push, NLP classification, workflow engine) but the "conversational Gosuto authoring" layer connecting them isn't built. Tracked as R17 + future Codex integration.

### 9. Auditable Inter-Agent Communication ✅ Complete (10/10)

`matrix.send_message` is policy-gated. Every send logged with trace ID. Audit breadcrumbs posted to admin room. Messaging targets declared in Gosuto and enforced by policy. Rate limiting prevents spam.

### 10. Updatable Plans ⚠️ Partial (4/10)

**Works:** Update cron schedules via command/NLP, update topology via deterministic commands, push new Gosuto versions, diff and rollback.

**Doesn't work yet:**
- "Change Kumo to do an exact search instead of a plan" requires manual YAML editing.
- No "show me the current plan for Kumo in plain English" command.
- No automatic downstream Gosuto re-rendering when a plan element changes.

**Infrastructure is ready** (versions, diffs, rollback, ACP push); the conversational editing UX isn't.

### 11. Any-Workflow Flexibility ⚠️ Partial (6/10)

**Works:** Generic workflow engine (`parse_input`, `tool`, `summarize`, `plan`, `send_message`, `persist`, `for_each`), multiple protocols per agent, generic templates, unlimited agent count.

**Doesn't work yet:**
- **Branching** (`if/else`) defined in type system but not implemented in runner.
- **Cross-agent workflow state tracking** — no orchestration layer for multi-agent pipeline state/failure recovery.
- **Durable persistence** — `persist` step is in-memory per run, not durable storage.
- **Novel workflow authoring is manual** — must write Gosuto YAML.

## Example Flows — Can They Work Today?

### Example 1: "Every morning, get me news about OpenAI"

**Yes, this works today.** Canonical Saito → Kumo flow. NLP decomposes the request, provisions both agents, wires topology. Only manual step: enter the Brave API key via the Kuze one-time link (by design). Validated by `make test-canonical-workflow-live-compose`.

### Example 2: "Every day, extract a business idea using an LLM, keep history"

**Partially.** Would need: (1) a Gosuto template for the "idea agent" with instructions, workflow protocol, summarize + send_message steps; (2) a persistence mechanism (MCP database server or accept Matrix chat as the history). User would have to write ~50 lines of YAML and run provisioning commands. **Why it can't be conversational yet:** R17 (template variable extraction) and conversational Gosuto authoring layer aren't built.

## Security Posture vs. OpenClaw

| Concern | OpenClaw | Ruriko |
|---------|----------|--------|
| Tool access | Unrestricted by default | First-match-wins policy, default deny |
| Self-modification | Agent can modify itself | Gitai binary is immutable |
| Secret exposure | Secrets in chat/config | Kuze one-time links only; never in Matrix, logs, or ACP |
| Runaway actions | No guardrails | Approval gates, rate limits, circuit breaker (R7 planned) |
| Audit trail | Limited | Every action traced with ID; breadcrumbs in Matrix |
| LLM as control plane | LLM decides everything | LLM translates; deterministic router decides |
| Scope creep | Full access to everything given | Scoped per-agent capabilities in Gosuto |

The remaining hardening (R7: circuit breakers, MCP restart backoff, Prometheus) is polish, not architecture.

## Where the Gap Really Is

The gap isn't security, architecture, or runtime. Those are done and solid.

The gap is **making the secure infrastructure feel effortless to a non-technical user:**

### Gap 1: Conversational Gosuto Authoring

Creating a new type of agent requires writing YAML. The bridge:
- **R17** — template variables at provision time (customise existing templates via flags/NLP)
- **Gosuto generation from description** — LLM takes intent + schema → produces valid config (Codex integration, post-MVP)
- **Conversational template editing** — NL diff → Gosuto patch → validate → version → push (R17.4, now tracked)

### Gap 2: Workflow Engine Expressiveness

- **Branching** (`if/else`) — defined in types, not in runner (now tracked in R6.2)
- **Cross-agent state tracking** — no orchestration layer (now tracked post-MVP)
- **Durable persistence** — `persist` is in-memory only (now tracked post-MVP)

### Gap 3: Plan Lifecycle Management

Mechanical parts exist (versioning, diffs, rollback, ACP push). Missing:
- "Show me what Kumo does" → plain-English plan readback (now tracked in R17.4)
- "Change step 3 to do X" → conversational Gosuto modification (now tracked in R17.4)
- Automatic re-rendering from changed constraints (now tracked in R17.4)

## Bottom Line

The hard part is done: security architecture, deterministic control, policy engine, immutable runtime, secret isolation — all working and well-tested. Most agent frameworks never get here.

What remains is the "last mile" UX: making all that security infrastructure invisible behind natural conversation. For Example 1 (daily news), the system works today. For Example 2 (daily business ideas), ~50 lines of YAML and a few commands. The vision is that even Example 2 should be a single conversation. That's the target.
